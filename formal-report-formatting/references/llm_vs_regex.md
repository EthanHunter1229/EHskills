# 标题识别方案对比：正则 vs LLM

## 问题背景

Word 文档的标题识别是排版自动化的核心挑战。传统方案依赖正则表达式匹配固定模式（如"第X节"、"一、"），但现实中文档格式千变万化。

---

## 方案对比

### 方案 A：正则匹配（当前默认）

#### 实现逻辑
```python
# 匹配"第X节"
pattern_h1 = r"^第[一二三四五六七八九十百零]+[节章]"

# 匹配"一、二、三、"
pattern_h2 = r"^[一二三四五六七八九十百零]+、"

# 匹配"（一）（二）"
pattern_h3 = r"^[（(][一二三四五六七八九十百零]+[)）]"

# 匹配"1. 2. 3."
pattern_h4 = r"^\d{1,2}[\.\、]\s*[\u4e00-\u9fa5]"
```

#### 优点
- ✅ **速度极快** - 毫秒级，适合大文档
- ✅ **规则明确** - 用户可自定义正则
- ✅ **无依赖** - 不需要网络/API 调用
- ✅ **确定性** - 同样输入永远相同输出

#### 缺点
- ❌ **格式僵化** - 只能识别预定义的编号模式
- ❌ **误判风险高**
  - "2021年公司营收..." 被误判为四级标题
  - "100.5万元投资额..." 被误判为标题
  - "3.14亿元净利润..." 被误判为标题
- ❌ **无法适应变化**
  - "Chapter 1"、"Part A" 无法识别
  - "核心竞争力"（无编号）无法识别
  - 自定义编号（如"S1"、"A.1"）无法识别
- ❌ **无语义理解**
  - 看不出"风险提示"是标题，"公司于2020年成立"是正文
  - 无法判断"第一节"和"第一章"哪个层级更高

#### 典型失败案例
```
✅ 正确识别：
"第一节 公司基本情况" → h1
"一、主营业务" → h2
"（一）产品介绍" → h3

❌ 误判：
"2021年公司实现营收100亿元..." → 误判为 h4（因为以"2021"开头）
"3.14亿元的净利润..." → 误判为 h4（因为"3."）
"100.5万元投资..." → 误判为 h4（因为"100."）

❌ 漏判：
"核心竞争力" → 无法识别（没有编号）
"Chapter 3: Market Analysis" → 无法识别（英文）
"风险提示" → 无法识别（没有编号）
```

### 方案 B：LLM 语义识别（推荐）

#### 实现逻辑
```python
# 提取文档段落上下文
paragraphs = [
    "[0] 招股说明书",
    "[1] 第一节 公司基本情况",
    "[2] 公司成立于2020年，主要从事...",
    "[3] 一、主营业务",
    "[4] 公司的主营业务包括...",
    ...
]

# 让 LLM 批量分析
prompt = f"""
分析以下文档段落，判断每个段落的类型：

{paragraphs}

为每段分配标签：
- cover: 封面标题
- h1/h2/h3/h4: 各级标题
- body: 正文

标题特征：简短、总结性、概括性
正文特征：较长、叙述性、描述性

输出 JSON:
[
  {"index": 0, "label": "cover"},
  {"index": 1, "label": "h1"},
  {"index": 2, "label": "body"},
  ...
]
"""

response = llm_call(prompt)  # 调用 Claude/GPT
classification = parse_json(response)
```

#### 优点
- ✅ **理解语义** - 根据内容性质判断，不依赖格式
- ✅ **适应任意格式**
  - "核心竞争力"、"风险提示" 能识别为标题
  - "Chapter 3"、"Part A" 能识别
  - "2021年营收情况" 能识别为标题（虽然以年份开头）
- ✅ **上下文感知** - 结合前后段落判断层级
- ✅ **跨语言支持** - 中英文混排文档也能处理
- ✅ **自我纠正** - 看到不合理层级跳跃会自动调整
- ✅ **可解释** - LLM 可以说明为什么某段是标题

#### 缺点
- ❌ **速度较慢** - 10-30秒（取决于文档长度）
- ❌ **需要 API** - 依赖 Claude/GPT 等外部服务
- ❌ **有 token 成本** - 每次调用消耗 token
- ❌ **非确定性** - 同样输入可能有微小差异（但不影响实际效果）

#### 典型成功案例
```
✅ 正确识别（正则无法做到）：
"核心竞争力" → h2（简短、总结性，虽然无编号）
"2021年营收情况" → h2（标题语境，不是正文叙述）
"Chapter 3: Market Analysis" → h1（英文标题）
"风险提示" → h2（典型的标题性短语）

✅ 正确排除（正则会误判）：
"2021年公司实现营收100亿元，同比增长..." → body（叙述性完整句子）
"3.14亿元的净利润主要来自..." → body（长句、描述性）
"100.5万元投资用于研发..." → body（叙述性）

✅ 层级判断：
"第一章 公司概述" → h1
  "一、基本信息" → h2
    "公司简介" → h3（LLM 能推断这是子标题）
    "公司历史" → h3
  "二、股权结构" → h2
```

## 混合方案（最佳实践）

结合两者优点：

```python
def hybrid_classify(paragraph, llm_result, regex_patterns):
    # 策略 1: LLM 优先（如果启用）
    if llm_result and paragraph.index in llm_result:
        return llm_result[paragraph.index]

    # 策略 2: 正则回退
    for level, pattern in regex_patterns.items():
        if re.match(pattern, paragraph.text):
            return level

    # 策略 3: Word 样式回退
    if paragraph.style == "Heading 1":
        return "h1"

    # 策略 4: 正文
    return "body"
```

**优势：**
- 标准文档用正则（快速）
- 非标准文档用 LLM（准确）
- 有保底逻辑（向后兼容）

## 使用建议

### 何时用正则模式
- 标准招股说明书、IPO 材料
- 严格遵循编号惯例的企业报告
- 大批量文档（速度优先）
- 离线环境（无网络）

```bash
python3 scripts/format_docx.py input.docx output.docx
```

### 何时用 LLM 模式
- 结构不规则的文档
- 无编号或自定义编号的文档
- 中英文混排文档
- 跨行业/跨类型文档（学术论文、合同、技术文档等）
- 准确性要求极高的场景

```bash
python3 scripts/format_docx.py input.docx output.docx --use-llm
```

## 成本对比

### 正则模式
- **时间成本**：< 1 秒
- **金钱成本**：0
- **维护成本**：需要为每种格式编写正则

### LLM 模式
- **时间成本**：10-30 秒
- **金钱成本**：~$0.01-0.05 / 文档（Claude Sonnet）
- **维护成本**：几乎为零（LLM 自适应）

### 建议
- 个人用户、小团队：优先 LLM（省人力）
- 大规模批处理：正则（省成本）
- 生产环境：混合模式（灵活性 + 保底）

## 技术实现细节

### LLM Prompt 设计要点

1. **明确任务**：分类标题 vs 正文，而非生成内容
2. **提供上下文**：给出前后段落，帮助判断层级
3. **结构化输出**：要求 JSON 格式，便于解析
4. **示例引导**：在 prompt 中给出典型示例
5. **错误处理**：LLM 返回格式错误时回退到正则

### 性能优化

1. **批量处理**：一次调用分析整个文档，而非逐段调用
2. **上下文截断**：超过 8000 字符时只送前半部分 + 提示后续省略
3. **缓存结果**：同一文档多次排版时复用 LLM 结果
4. **异步调用**：用户上传文档后立即发起 LLM 调用，排版时直接读取

## 未来改进方向

1. **本地模型**：支持 Ollama/LLaMA 等本地模型（无 API 成本）
2. **用户训练**：让用户提供示例文档，fine-tune 分类模型
3. **增量学习**：每次用户手动修正后，记录为训练样本
4. **混合模式自动选择**：根据文档特征自动决定用哪种模式
5. **可视化编辑**：生成 HTML 预览，用户点击调整标题层级

---

## 结论

**正则模式**适合标准化场景，**LLM 模式**适合复杂场景。两者结合使用是当前最佳方案。

长期来看，随着 LLM 推理成本下降（Claude Haiku、本地模型），**LLM 将成为默认选择**，正则仅作为离线场景的保底方案。
